\chapter{Veille technologique et sécurité}

\section{Pourquoi une veille technologique ?}

Les technologies évoluent vite. Ce qui était la meilleure pratique il y a un an peut être obsolète aujourd'hui, voire présenter des failles de sécurité. Pour que Destiny Raid Companion reste performant et sécurisé, je dois me tenir informé des évolutions.

\section{La sécurité au cœur du projet}

\subsection{Les protections que j'ai mises en place}

\begin{itemize}
    \item \textbf{Injection SQL :} J'utilise Prisma, donc les requêtes sont automatiquement paramétrées. Impossible d'injecter du SQL.
    \item \textbf{XSS (Cross-Site Scripting) :} React échappe automatiquement tout ce qui est affiché. Pour le contenu riche (guides), j'utilise DOMPurify pour nettoyer.
    \item \textbf{CSRF :} Je vérifie l'origine des requêtes et j'utilise des tokens pour les actions sensibles.
    \item \textbf{OAuth :} Je suis les recommandations de Bungie, avec PKCE pour plus de sécurité.
    \item \textbf{Rate limiting :} Je limite le nombre de requêtes par IP pour éviter les abus.
\end{itemize}

\subsection{Chiffrement des données sensibles}

Les tokens d'accès Bungie sont des données très sensibles. Je les chiffre avant de les stocker en base.

\begin{lstlisting}[language=JavaScript]
const crypto = require('crypto');

class TokenEncryptor {
  encryptBungieToken(token) {
    const algorithm = 'aes-256-gcm';
    const key = Buffer.from(process.env.ENCRYPTION_KEY, 'hex');
    const iv = crypto.randomBytes(16);
    
    const cipher = crypto.createCipheriv(algorithm, key, iv);
    let encrypted = cipher.update(token, 'utf8', 'hex');
    encrypted += cipher.final('hex');
    
    return {
      encrypted,
      iv: iv.toString('hex'),
      authTag: cipher.getAuthTag().toString('hex')
    };
  }
}
\end{lstlisting}

\subsection{Validation des entrées}

Tout ce qui vient de l'utilisateur est validé avant d'être traité.

\begin{lstlisting}[language=JavaScript]
const Joi = require('joi');

const raidGuideSchema = Joi.object({
  title: Joi.string().max(200).required(),
  description: Joi.string().max(5000),
  difficulty: Joi.string().valid('novice', 'normal', 'master'),
  steps: Joi.array().items(
    Joi.object({
      title: Joi.string().max(100).required(),
      description: Joi.string().max(1000).required()
    })
  ).min(1).max(20)
});

function validateRaidGuide(guideData) {
  const { error, value } = raidGuideSchema.validate(guideData);
  if (error) {
    throw new ValidationError(error.details);
  }
  return value;
}
\end{lstlisting}

\subsection{Plan de réponse aux incidents}

Si quelque chose tourne mal, j'ai un plan :

\begin{enumerate}
    \item \textbf{Détection :} Une alerte m'informe du problème (email, Slack).
    \item \textbf{Analyse :} Je consulte les logs pour comprendre ce qui se passe.
    \item \textbf{Confinement :} Si nécessaire, je passe en mode maintenance.
    \item \textbf{Correction :} Je déploie un correctif.
    \item \textbf{Retour d'expérience :} J'écris un post-mortem pour ne pas refaire la même erreur.
\end{enumerate}

\section{Ce que j'ai appris}

\begin{itemize}
    \item \textbf{La veille, ce n'est pas optionnel :} Les failles de sécurité sont découvertes tous les jours. Si je n'écoute pas, je peux laisser une porte ouverte.
    \item \textbf{La sécurité, ça se pense dès le début :} Ajouter de la sécurité après coup est beaucoup plus difficile que de la concevoir dès le départ.
    \item \textbf{Les outils aident, mais ne font pas tout :} SonarQube, Snyk, npm audit sont utiles, mais ils ne remplacent pas la réflexion.
    \item \textbf{Documenter ses incidents :} Chaque incident est une occasion d'apprendre. J'écris toujours un petit post-mortem pour moi-même.
\end{itemize}
