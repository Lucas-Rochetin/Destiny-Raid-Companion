\chapter{Sécurité applicative et RGPD}

\section{Pourquoi la sécurité est cruciale pour ce projet}

Une plateforme qui gère des comptes utilisateur et des données personnelles doit être sécurisée. C'est à la fois une obligation légale (RGPD) et une nécessité pour la confiance des utilisateurs. Si un joueur perd l'accès à son compte ou si ses données sont volées, c'est fini : il ne reviendra pas.

Dans ce chapitre, je détaille les mesures que j'ai prises pour :
\begin{itemize}
    \item Protéger l'application contre les attaques courantes (OWASP Top 10).
    \item Sécuriser l'authentification et les données sensibles.
    \item Être conforme au RGPD (respect de la vie privée des joueurs).
\end{itemize}

\section{Protection contre les vulnérabilités OWASP}

J'ai suivi les recommandations de l'OWASP (Open Web Application Security Project) pour éviter les failles les plus courantes.

\subsection{Injection SQL (requêtes paramétrées)}

Le risque : Un attaquant pourrait insérer du code SQL dans un champ de formulaire pour manipuler la base de données.

Comment je m'en protège :
\begin{itemize}
    \item Je n'écris jamais de requêtes SQL "à la main". J'utilise \textbf{Prisma}, un ORM qui génère automatiquement des requêtes paramétrées.
    \item Prisma échappe automatiquement les données avant de les envoyer à PostgreSQL.
    \item Même si un utilisateur saisit ``'; DROP TABLE users; --`` dans un champ, Prisma le traite comme une chaîne de caractères, pas comme du code SQL.
\end{itemize}

\subsection{XSS (Cross-Site Scripting)}

Le risque : Un attaquant pourrait injecter du code JavaScript malveillant dans un guide ou un commentaire. Quand un autre utilisateur consulterait la page, ce code s'exécuterait.

Comment je m'en protège :
\begin{itemize}
    \item \textbf{React échappe automatiquement} tout ce qui est affiché. Si je mets \texttt{\{userInput\}} dans mon JSX, React transforme les caractères spéciaux en entités HTML.
    \item Pour les guides qui contiennent du HTML (formatage, images), j'utilise \textbf{DOMPurify} pour nettoyer le contenu avant de l'afficher. Je n'autorise que certaines balises (p, strong, em, ul, li, h2, h3) et j'interdis tout attribut dangereux (comme \texttt{onclick}).
\end{itemize}

\begin{lstlisting}[language=JavaScript]
// Nettoyage du contenu des guides avant affichage
const sanitizeHtml = (html) => {
  return DOMPurify.sanitize(html, {
    ALLOWED_TAGS: ['p', 'strong', 'em', 'ul', 'li', 'h2', 'h3'],
    ALLOWED_ATTR: ['class']  // Seulement les classes CSS, pas les événements JS
  });
};
\end{lstlisting}

\subsection{CSRF (Cross-Site Request Forgery)}

Le risque : Un attaquant pourrait amener un utilisateur connecté à exécuter une action sans son consentement (par exemple, supprimer une escouade) en lui faisant visiter un site malveillant.

Comment je m'en protège :
\begin{itemize}
    \item J'utilise des tokens CSRF pour les actions sensibles.
    \item Les requêtes vers l'API doivent inclure un token spécial, stocké dans un cookie sécurisé et vérifié par le backend.
    \item Je vérifie aussi l'en-tête \texttt{Origin} ou \texttt{Referer} pour m'assurer que la requête vient bien de mon application.
\end{itemize}

\subsection{Faille d'authentification}

Le risque : Un attaquant pourrait voler des tokens JWT ou forcer l'accès à des comptes.

Comment je m'en protège :
\begin{itemize}
    \item Les tokens JWT ont une durée de vie \textbf{courte} (1 heure). Après ce délai, l'utilisateur doit se réauthentifier ou utiliser un refresh token.
    \item Les refresh tokens sont stockés dans Redis avec une durée de vie plus longue (30 jours), mais ils sont à usage unique.
    \item Toutes les communications sont en \textbf{HTTPS} (chiffrement TLS). Les tokens ne circulent jamais en clair.
    \item Je limite le nombre de tentatives de connexion : 10 par 15 minutes. Au-delà, l'adresse IP est temporairement bloquée.
\end{itemize}

\begin{lstlisting}[language=JavaScript]
// Rate limiting pour l'authentification
const authLimiter = rateLimit({
  windowMs: 15 * 60 * 1000,  // 15 minutes
  max: 10,                    // 10 tentatives max
  message: 'Trop de tentatives, réessayez dans 15 minutes'
});
app.use('/api/auth/login', authLimiter);
\end{lstlisting}

\subsection{Headers de sécurité}

J'utilise le middleware \texttt{helmet} pour ajouter automatiquement des en-têtes HTTP sécurisés :

\begin{itemize}
    \item \textbf{CSP (Content Security Policy) :} Je spécifie d'où peuvent venir les ressources (scripts, styles, images). Par exemple, les scripts ne peuvent venir que de mon domaine et de Bungie.net.
    \item \textbf{HSTS :} Je force le navigateur à toujours utiliser HTTPS pour mon domaine.
    \item \textbf{X-Frame-Options :} J'interdis l'affichage de mon site dans une iframe (pour éviter le clickjacking).
\end{itemize}

\begin{lstlisting}[language=JavaScript]
const helmet = require('helmet');

app.use(helmet({
  contentSecurityPolicy: {
    directives: {
      defaultSrc: ["'self'", "https://www.bungie.net"],
      scriptSrc: ["'self'", "https://www.bungie.net"],
      imgSrc: ["'self'", "https://www.bungie.net"]
    }
  },
  hsts: {
    maxAge: 31536000,  // 1 an
    includeSubDomains: true,
    preload: true
  }
}));
\end{lstlisting}

\section{Authentification et autorisation}

\subsection{Le flux OAuth avec Bungie}

Je n'invente pas mon propre système d'authentification. Je délègue à Bungie via OAuth 2.0. Voici comment ça fonctionne :

\begin{enumerate}
    \item L'utilisateur clique sur "Se connecter avec Bungie".
    \item Il est redirigé vers le site de Bungie, où il saisit ses identifiants.
    \item Bungie demande à l'utilisateur s'il autorise mon application à accéder à ses données (pseudo, statistiques, personnages).
    \item Si l'utilisateur accepte, Bungie redirige vers mon application avec un code temporaire.
    \item Mon backend échange ce code contre un token d'accès Bungie (valable 1 heure) et un refresh token.
    \item Je crée ou mets à jour l'utilisateur dans ma base, et je lui génère un token JWT interne pour les futures requêtes.
\end{enumerate}

\subsection{Gestion sécurisée des tokens}

Les tokens Bungie sont très sensibles : ils donnent accès aux données du joueur. Je les protège ainsi :

\begin{itemize}
    \item Ils sont \textbf{chiffrés} en base de données avec AES-256 avant stockage.
    \item La clé de chiffrement n'est pas dans le code source. Elle est dans les variables d'environnement, et tournée tous les 90 jours.
    \item Les tokens ne sont \textbf{jamais envoyés au frontend}. Seul le JWT interne circule entre le frontend et le backend.
\end{itemize}

\begin{lstlisting}[language=JavaScript]
// Chiffrement d'un token avant stockage
const encryptToken = (token) => {
  const iv = crypto.randomBytes(16);
  const cipher = crypto.createCipheriv('aes-256-gcm', encryptionKey, iv);
  
  let encrypted = cipher.update(JSON.stringify(token), 'utf8', 'hex');
  encrypted += cipher.final('hex');
  const authTag = cipher.getAuthTag();
  
  return {
    encrypted,
    iv: iv.toString('hex'),
    authTag: authTag.toString('hex')
  };
};
\end{lstlisting}

\subsection{Système de rôles et permissions}

Tous les utilisateurs n'ont pas les mêmes droits. J'ai défini plusieurs rôles :

\begin{itemize}
    \item \textbf{PLAYER :} Peut consulter les guides, créer une escouade, rejoindre des sessions.
    \item \textbf{SQUAD\_LEADER :} Peut en plus inviter/kicker des membres, modifier les sessions de son escouade.
    \item \textbf{GUIDE\_WRITER :} Peut créer et modifier des guides (après validation).
    \item \textbf{MODERATOR :} Peut bannir des utilisateurs, modérer les contenus.
    \item \textbf{ADMIN :} Tous les droits, y compris la configuration système.
\end{itemize}

Chaque requête sensible vérifie que l'utilisateur a la permission nécessaire :

\begin{lstlisting}[language=JavaScript]
const requirePermission = (permission) => {
  return (req, res, next) => {
    const userRole = req.user.role;
    const permissions = ROLES[userRole] || [];
    
    if (!permissions.includes(permission)) {
      return res.status(403).json({ 
        error: 'Permission refusée' 
      });
    }
    next();
  };
};

// Utilisation dans une route
router.post('/guides', 
  authenticateToken,
  requirePermission('GUIDE_CREATE'),
  guideController.createGuide
);
\end{lstlisting}

\section{Conformité RGPD}

Le RGPD n'est pas une option. J'ai conçu l'application pour respecter la vie privée des utilisateurs dès le départ (privacy by design).

\subsection{Quelles données je collecte et pourquoi}

\begin{center}
\begin{tabular}{|p{4cm}|p{6cm}|}
\hline
\textbf{Donnée} & \textbf{Pourquoi je la collecte} \\
\hline
Bungie ID & Identifier l'utilisateur de manière unique \\
\hline
Nom d'affichage & Afficher le pseudo dans les escouades, les guides \\
\hline
Tokens d'accès Bungie & Récupérer les données de jeu (personnages, stats) \\
\hline
Historique des raids & Suivre la progression, afficher des badges \\
\hline
Messages d'escouade & Permettre la communication entre joueurs \\
\hline
Logs de connexion & Sécurité (détecter les accès suspects) \\
\hline
\end{tabular}
\end{center}

\subsection{Base légale du traitement}

Pour chaque donnée, j'ai une base légale :
\begin{itemize}
    \item \textbf{Consentement :} L'utilisateur accepte explicitement en se connectant avec Bungie.
    \item \textbf{Exécution du contrat :} Certaines données sont nécessaires au fonctionnement du service (ex: les membres d'une escouade).
    \item \textbf{Intérêt légitime :} Les logs de sécurité sont nécessaires pour protéger l'application.
\end{itemize}

\subsection{Droits des utilisateurs}

Le RGPD donne des droits aux utilisateurs. J'ai implémenté les moyens de les exercer :

\begin{itemize}
    \item \textbf{Droit d'accès :} L'utilisateur peut exporter toutes ses données depuis son profil (format JSON ou CSV).
    \item \textbf{Droit de rectification :} Il peut modifier ses informations personnelles (sauf le Bungie ID, qui est lié à son compte).
    \item \textbf{Droit à l'effacement :} Il peut demander la suppression de son compte. Je supprime ou anonymise toutes ses données.
    \item \textbf{Droit d'opposition :} Il peut refuser certains traitements (ex: analytics).
    \item \textbf{Portabilité :} L'export des données est dans un format standard, réutilisable ailleurs.
\end{itemize}

\begin{lstlisting}[language=JavaScript]
// API d'export des données utilisateur
app.get('/api/user/export', authenticateToken, async (req, res) => {
  const userData = {
    profile: await userService.getProfile(req.user.id),
    squads: await squadService.getUserSquads(req.user.id),
    sessions: await sessionService.getUserSessions(req.user.id),
    badges: await badgeService.getUserBadges(req.user.id)
  };
  
  res.json({
    exportDate: new Date().toISOString(),
    data: userData,
    format: 'RGPD-compliant'
  });
});
\end{lstlisting}

\subsection{Durée de conservation}

Je ne garde pas les données indéfiniment :

\begin{center}
\begin{tabular}{|p{4cm}|p{3cm}|p{4cm}|}
\hline
\textbf{Type de donnée} & \textbf{Durée} & \textbf{Justification} \\
\hline
Données de connexion & 3 ans & Sécurité (détection d'intrusion) \\
\hline
Messages d'escouade & 2 ans & Vie privée des conversations \\
\hline
Guides créés & Indéfinie & Contribution publique \\
\hline
Statistiques de jeu & 5 ans & Historique de progression \\
\hline
\end{tabular}
\end{center}

Après ces durées, les données sont anonymisées ou supprimées.

\subsection{Sécurité technique des données}

\begin{itemize}
    \item \textbf{Chiffrement au repos :} Les données sensibles (tokens) sont chiffrées en base avec AES-256.
    \item \textbf{Chiffrement en transit :} HTTPS partout, TLS 1.3.
    \item \textbf{Accès :} Principe du moindre privilège. Le compte de la base de données n'a que les droits nécessaires.
    \item \textbf{Sauvegardes :} Quotidiennes, chiffrées, stockées dans une région différente.
    \item \textbf{Logs d'accès :} Toute consultation de données sensibles est loguée et auditable.
\end{itemize}

\section{Monitoring et détection d'incidents}

La sécurité, ce n'est pas que de la prévention. C'est aussi de la détection et de la réaction.

\subsection{Ce que je surveille}

\begin{itemize}
    \item \textbf{Tentatives de connexion échouées :} Si un utilisateur échoue 10 fois, je bloque temporairement et j'envoie une alerte.
    \item \textbf{Requêtes anormales :} Trop de requêtes d'un même IP, accès à des endpoints sensibles, etc.
    \item \textbf{Erreurs applicatives :} Certaines erreurs peuvent cacher des tentatives d'exploitation.
    \item \textbf{Changements dans les tokens Bungie :} Si un refresh token est utilisé alors qu'il ne devrait pas, c'est peut-être un vol.
\end{itemize}

\subsection{Plan de réponse à incident}

Si un incident de sécurité survient (ex: compte compromis, faille découverte), j'ai un plan :

\begin{enumerate}
    \item \textbf{Détection :} Alerte reçue (logs, utilisateur qui signale).
    \item \textbf{Analyse :} Comprendre ce qui s'est passé, quelles données sont touchées.
    \item \textbf{Confinement :} Bloquer l'attaque (bannir IP, révoquer tokens, désactiver compte).
    \item \textbf{Récupération :} Restaurer les données si nécessaire, corriger la faille.
    \item \textbf{Notification :} Informer les utilisateurs concernés et la CNIL si nécessaire (obligation légale dans les 72h).
    \item \textbf{Post-mortem :} Analyser pourquoi c'est arrivé et comment éviter que ça se reproduise.
\end{enumerate}

\section{Ce que j'ai appris sur la sécurité}

Travailler sur ce projet m'a fait prendre conscience que la sécurité, ce n'est pas une fonctionnalité qu'on ajoute à la fin. C'est quelque chose qu'on doit penser dès la conception.

Quelques principes simples mais efficaces :
\begin{itemize}
    \item \textbf{Ne jamais faire confiance aux entrées utilisateur.} Tout ce qui vient du frontend doit être validé et nettoyé.
    \item \textbf{Moins on a de données sensibles, mieux c'est.} Je ne stocke que ce dont j'ai vraiment besoin.
    \item \textbf{Les failles les plus graves sont souvent les plus simples.} Un oubli de validation, un message d'erreur trop bavard, un token en clair dans une URL...
    \item \textbf{Les audits et les tests sont essentiels.} Je ne peux pas penser à tout tout seul. Les outils d'analyse (Snyk, npm audit) m'aident à détecter ce que j'aurais pu rater.
\end{itemize}
