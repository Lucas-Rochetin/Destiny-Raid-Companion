\chapter{Méthodologie et organisation}

\section{Comment j'organise mon travail avec GitHub}

Pour ce projet en solo, j'avais besoin d'une méthode qui me permette à la fois de garder le cap sur mes objectifs et de m'adapter quand je rencontre un imprévu (par exemple, un changement dans l'API Bungie). J'ai choisi GitHub pour tout gérer : le code, bien sûr, mais aussi le suivi des tâches, la planification et la documentation.

\subsection{Pourquoi j'ai choisi une méthode Agile... adaptée au solo}

La méthode Agile est habituellement pensée pour des équipes, mais elle m'a paru parfaitement adaptée à mon contexte pour plusieurs raisons :

\begin{itemize}
    \item \textbf{Le projet est innovant :} Je ne sais pas à l'avance exactement comment les joueurs vont utiliser l'outil. Je dois pouvoir m'adapter à leurs retours.
    \item \textbf{L'API Bungie est complexe :} Elle peut évoluer, ou je peux découvrir des limitations en cours de route. Il faut que je puisse ajuster mon planning.
    \item \textbf{Je suis seul :} Cela me donne une grande flexibilité pour réagir vite, mais ça m'oblige aussi à être très discipliné pour ne pas me disperser.
\end{itemize}

\subsection{Mon adaptation de GitFlow}

Même seul, j'utilise GitFlow (ou une version simplifiée). Pourquoi ? Parce que ça m'oblige à être propre :

\begin{itemize}
    \item Je crée une \texttt{feature/ma-fonctionnalité} pour chaque nouvelle fonction.
    \item Je prépare les versions dans des branches \texttt{release/v1.0}.
    \item Si un bug critique apparaît en production, je le corrige dans une branche \texttt{hotfix/...} sans toucher au développement en cours.
\end{itemize}

Cette discipline peut sembler lourde pour un projet solo, mais elle me prépare à l'éventualité d'avoir des contributeurs plus tard. Surtout, elle me permet de revenir en arrière facilement si une modification pose problème.

\subsection{Mes "rituels" Agiles, version solo}

Même sans réunion d'équipe, j'ai adapté les rituels Agiles pour garder le rythme :

\begin{center}

\end{center}

\subsection{Ce que me disent mes métriques}

Grâce à GitHub, je peux suivre quelques indicateurs qui m'aident à m'améliorer :

\begin{itemize}
    \item \textbf{Vélocité stable (8-12 points par sprint) :} Je suis régulier, je tiens mes engagements.
    \item \textbf{Taux de bugs (15\%) :} C'est un peu élevé. Cela me dit que je dois écrire plus de tests avant de livrer.
    \item \textbf{Temps de cycle (4,2 jours) :} Une tâche met en moyenne 4 jours à être terminée. C'est correct, mais je pourrais découper mes tâches en plus petits morceaux pour aller plus vite.
    \item \textbf{Temps de lead réduit (de 7 à 5 jours) :} Je vais plus vite qu'au début pour passer de l'idée à la livraison. C'est bon signe.
\end{itemize}

\subsection{Mon tableau Kanban}

Mon tableau GitHub Projects est organisé comme une chaîne de montage, avec une limite de deux tâches en cours pour ne pas me disperser :

\begin{verbatim}
Backlog → Sprint Backlog → À faire → En cours (max 2) → Revue → Terminé
\end{verbatim}

Chaque colonne a un sens précis :
\begin{itemize}
    \item \textbf{Backlog :} Toutes les idées, toutes les fonctionnalités futures.
    \item \textbf{Sprint Backlog :} Celles que j'ai décidé de faire dans les deux prochaines semaines.
    \item \textbf{En cours :} Ce sur quoi je travaille maintenant (jamais plus de deux).
    \item \textbf{Revue :} Je vérifie que le code est propre et que ça marche.
    \item \textbf{Terminé :} La fonctionnalité est livrée et fonctionnelle.
\end{itemize}

\subsection{Comment j'estime mes tâches}

J'utilise les "story points", une échelle qui mesure la complexité, pas le temps exact. Ça m'aide à prévoir sans me perdre dans les détails :

\begin{itemize}
    \item \textbf{1 point :} Tâche simple, moins d'une journée (ex: corriger une faute d'orthographe).
    \item \textbf{3 points :} Tâche moyenne, 1-2 jours (ex: créer un composant React simple).
    \item \textbf{5 points :} Tâche complexe, 3-4 jours (ex: intégrer un appel API avec gestion d'erreurs).
    \item \textbf{8 points :} Tâche très complexe, plus de 5 jours (ex: créer un algorithme de matching entre joueurs).
\end{itemize}

Quand une fonctionnalité touche à l'API Bungie, j'ajoute automatiquement 2 points de plus, car je sais que c'est risqué et imprévisible.

\section{Mes règles pour un code propre (versioning et conventions)}

Travailler seul ne veut pas dire travailler n'importe comment. Je me suis fixé des règles claires pour que mon code reste lisible et maintenable.

\subsection{Le fichier CONTRIBUTING.md}

J'ai créé un fichier \texttt{CONTRIBUTING.md} à la racine du projet. Il explique à quelqu'un (ou à moi dans 6 mois) comment reprendre le projet :

\begin{itemize}
    \item Comment installer l'environnement (avec les clés API Bungie).
    \item Quelles sont les règles de code (ESLint, Prettier).
    \item Comment nommer ses branches et ses commits.
    \item Comment lancer les tests et ce qu'on attend comme couverture.
    \item La checklist à vérifier avant de proposer une modification.
\end{itemize}

\subsection{Comment je nomme mes branches}

Chaque type de modification a sa branche dédiée :

\begin{center}
\begin{tabular}{|p{3cm}|p{8cm}|}
\hline
\textbf{Type} & \textbf{Exemple de nom} \\
\hline
Nouvelle fonctionnalité & \texttt{feature/authentification-bungie} ou \texttt{feature/issue-123} \\
\hline
Correction de bug & \texttt{fix/erreur-connexion} ou \texttt{fix/issue-456} \\
\hline
Correctif urgent & \texttt{hotfix/api-bungie-cassee} \\
\hline
Préparation d'une version & \texttt{release/v1.0.0} \\
\hline
Documentation & \texttt{docs/guide-installation} \\
\hline
\end{tabular}
\end{center}

\subsection{Comment je nomme mes commits}

J'utilise la convention "Conventional Commits". Ça peut sembler technique, mais ça rend l'historique du projet très clair :

\begin{lstlisting}
feat: ajout de l'authentification OAuth Bungie
fix: correction du rafraîchissement du token expiré
docs: mise à jour du README avec les variables d'environnement
test: ajout de tests pour le service de gestion des escouades
refactor: simplification du composant GuideViewer
style: formatage du code avec Prettier
chore: mise à jour des dépendances
\end{lstlisting}

\subsection{Ma "Definition of Done"}

Avant de marquer une tâche comme terminée, je vérifie cette liste :

\begin{itemize}
    \item Le code est testé (couverture > 80\%).
    \item J'ai fait une relecture de mon propre code (auto-review).
    \item La pipeline GitHub Actions passe (build, tests, analyse de sécurité).
    \item La fonctionnalité est déployée sur mon environnement de test et je l'ai validée manuellement.
    \item La documentation est à jour (commentaires dans le code, éventuellement README).
    \item Il n'y a pas de vulnérabilité de sécurité évidente.
\end{itemize}

\section{Comment je planifie et je suis mon avancement}

\subsection{Un outil, deux vues}

GitHub me permet d'avoir deux visions complémentaires :

\begin{itemize}
    \item \textbf{La roadmap} (vue macro) : Je vois les grandes étapes sur plusieurs mois. C'est ce que je montre aux utilisateurs pour leur donner une idée de l'avenir du projet.
    \item \textbf{Le tableau Kanban} (vue micro) : Je vois ce que je dois faire aujourd'hui et cette semaine. C'est mon outil de pilotage quotidien.
\end{itemize}

\subsection{Extrait de ma roadmap}

\begin{verbatim}
Phase 1 : MVP (oct. 2025 - mars 2026)
├── Sprint 1 : Authentification et base de données (4 semaines)
├── Sprint 2 : Guides interactifs et gestion d'escouades (6 semaines)
└── Sprint 3 : Calendrier et profils (4 semaines)

Phase 2 : Version 1.1 (avril - mai 2026)
├── Sprint 4 : Améliorations UX et corrections
└── Sprint 5 : Déploiement final
\end{verbatim}

\subsection{Le lien entre user stories, tests et jalons}

Je m'assure que tout est connecté dans GitHub :

\begin{enumerate}
    \item Je crée une \textbf{user story} (par exemple : "En tant que joueur, je veux me connecter avec Bungie").
    \item Je lui associe des \textbf{tests} (des issues "Test : connexion réussie", "Test : token expiré").
    \item Je la rattache à un \textbf{milestone} (par exemple "MVP v1.0").
\end{enumerate}

Comme ça, quand je termine une fonctionnalité, je sais exactement ce qui a été testé et pour quelle version c'est prévu.

\section{Ce que je retire de cette organisation}

Au début, j'avais tendance à vouloir tout faire en même temps et à noter mes idées sur des Post-It. Avec cette organisation GitHub, j'ai gagné en clarté et en régularité. Je sais toujours où j'en suis, ce qui est prévu pour demain, et ce qui est vraiment important.

En résumé, cette méthode me sert à :
\begin{itemize}
    \item Garder le cap sur mes objectifs malgré les imprévus techniques.
    \item Travailler de manière plus professionnelle, comme si j'étais en équipe.
    \item Apprendre de mes erreurs en mesurant objectivement ma progression.
\end{itemize}
