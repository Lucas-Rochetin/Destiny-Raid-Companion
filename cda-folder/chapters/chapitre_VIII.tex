\chapter{Déploiement et intégration continue (CI/CD)}

\section{Pourquoi automatiser le déploiement ?}

Quand on développe seul, on peut être tenté de déployer "à la main" : on fait un changement, on copie les fichiers sur le serveur, on redémarre... Mais ça devient vite source d'erreurs :
\begin{itemize}
    \item On oublie une étape.
    \item On ne teste pas la même chose que ce qui part en production.
    \item Revenir en arrière est compliqué.
\end{itemize}

J'ai donc mis en place une chaîne d'intégration et de déploiement continus (CI/CD) pour que tout soit automatisé, testé, et reproductible.

\section{Containerisation avec Docker}

Pour que l'application tourne de la même manière sur ma machine, sur le serveur de test, et en production, j'utilise Docker.

\subsection{Le backend (Node.js)}

\begin{lstlisting}[language=dockerfile]
FROM node:20-alpine

WORKDIR /app

COPY package*.json ./
RUN npm ci
COPY . .

EXPOSE 3002
CMD ["node" , "server.js"]
\end{lstlisting}

Ce fichier dit simplement :
\begin{itemize}
    \item On part d'une image Node.js officielle.
    \item On installe les dépendances.
    \item On copie le code.
    \item On expose le port 3000.
    \item On lance l'application.
\end{itemize}

\subsection{Le frontend (HTML/CSS/JS)}

Pour le frontend, pas besoin de Node.js en production. Un simple serveur Nginx suffit :

\begin{lstlisting}[language=dockerfile]
FROM nginx:alpine
COPY ./public /usr/share/nginx/html
EXPOSE 80
\end{lstlisting}

\subsection{Orchestration avec Docker Compose}

Pour lancer tous les services ensemble, j'utilise Docker Compose :

\begin{lstlisting}[language=yaml]
version: "3.8"

services:
  frontend:
    build: ./frontend
    ports:
      - "8080:80"
    depends_on:
      - backend

  backend:
    build: ./backend
    ports:
      - "3000:3000"
    environment:
      - NODE_ENV=production
      - BUNGIE_API_KEY=${BUNGIE_API_KEY}
    # La clé API est dans un fichier .env, pas dans le code !
\end{lstlisting}

Avec une seule commande, toute l'application démarre :

\begin{verbatim}
docker-compose up -d
\end{verbatim}

\section{Gestion des secrets}

Les données sensibles (clé API Bungie) ne doivent jamais être dans le code source. Je les passe par variables d'environnement :

\begin{itemize}
    \item En local : dans un fichier \texttt{.env} (ignoré par Git).
    \item Sur GitHub : dans les "secrets" du dépôt.
    \item En production : dans l'environnement du serveur.
\end{itemize}

\section{Intégration continue (CI) avec GitHub Actions}

À chaque fois que je pousse du code, GitHub Actions lance automatiquement une série de vérifications.

\subsection{Ma pipeline CI}

\begin{lstlisting}[language=yaml]
name: Deploy de Production

on:
  push:
    branches: [ main ]
  pull_request:
    branches: [ main ]

jobs:

  test:
    runs-on: ubuntu-latest

    steps:
      - name: Récupération du repo
        uses: actions/checkout@v4

      - name: Installer Node
        uses: actions/setup-node@v4
        with:
          node-version: 20

      - name: Installer dépendances backend
        run: |
          cd app/backend
          npm install

      - name: Installer dépendances frontend
        run: |
          cd app/frontend
          npm install

      - name: Lancer les tests
        run: |
          cd app/backend
          npm run test --if-present

  docker:

    needs: test
    runs-on: ubuntu-latest

    steps:
      - name: Récupération du repo
        uses: actions/checkout@v4

      - name: Login DockerHub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKER_USERNAME }}
          password: ${{ secrets.DOCKER_PASSWORD }}

      - name: Build images
        run: |
            cd app
            docker compose build

      - name: Push images
        run: |
          docker push ${{ secrets.DOCKER_USERNAME }}/destiny-raid-companion-backend:latest
          docker push ${{ secrets.DOCKER_USERNAME }}/destiny-raid-companion-frontend:latest

  deploy:

    needs: docker
    runs-on: ubuntu-latest

    steps:

      - name: Deploy sur EC2
        uses: appleboy/ssh-action@v1.0.3
        with:
          host: ${{ secrets.EC2_HOST }}
          username: ${{ secrets.EC2_USER }}
          key: ${{ secrets.EC2_SSH_KEY }}

          script: |
            docker pull ${{ secrets.DOCKER_USERNAME }}/destiny-raid-companion-backend:latest
            docker pull ${{ secrets.DOCKER_USERNAME }}/destiny-raid-companion-frontend:latest

            cd destiny/destiny/app/

            docker compose down
            docker compose up -d
\end{lstlisting}

\subsection{Ce que ça vérifie}

\begin{itemize}
    \item Les dépendances s'installent correctement.
    \item Le code respecte les règles de formatage (linter).
    \item Tous les tests passent (unitaires et intégration).
\end{itemize}

Si une étape échoue, le pipeline s'arrête et je reçois une notification. Je ne peux pas merger du code qui casse quelque chose.

\section{Déploiement continu (CD)}

Quand le code est validé sur la branche principale, je déclenche le déploiement.

\subsection{Étapes du déploiement}

\begin{enumerate}
    \item Les images Docker sont reconstruites.
    \item Elles sont poussées vers un registre (Docker Hub ou GitHub Container Registry).
    \item Sur le serveur, je tire la nouvelle image et je redémarre.
\end{enumerate}

\subsection{Commandes de déploiement}

Le déploiement est automatisé à 100\% :

\begin{itemize}
    \item \textbf{Branche \texttt{develop} :} déploiement automatique sur l'environnement de \textbf{staging}
    \item \textbf{Branche \texttt{main} :} déploiement automatique sur \textbf{production}
\end{itemize}

Quand je merge une pull request sur \texttt{main}, GitHub Actions :
\begin{enumerate}
    \item Se connecte au serveur via SSH (clé privée stockée dans les secrets GitHub)
    \item Exécute \texttt{docker-compose pull} pour récupérer les dernières images
    \item Redémarre les conteneurs avec \texttt{docker-compose up -d}
    \item Nettoie les images anciennes
\end{enumerate}

Plus besoin de me connecter manuellement au serveur. La fiabilité est maximale.

\section{Monitoring et logs}

Une fois déployé, il faut surveiller que tout va bien.

\subsection{Logs applicatifs}

Les logs des conteneurs sont accessibles avec :

\begin{verbatim}
docker logs destiny-backend
docker logs destiny-frontend
\end{verbatim}

\subsection{Health checks}

Docker peut vérifier automatiquement que l'application est vivante :

\begin{lstlisting}[language=yaml]
services:
  backend:
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:3000/health"]
      interval: 30s
      timeout: 10s
      retries: 3
\end{lstlisting}

Si le health check échoue trop de fois, Docker redémarre le conteneur.

\section{Documentation pour reprendre le projet}

Si quelqu'un veut déployer mon application (ou moi dans 6 mois), j'ai tout documenté :

\subsection{Prérequis}
\begin{itemize}
    \item Docker et Docker Compose installés
    \item Une clé API Bungie valide
    \item Un fichier \texttt{.env} avec les variables nécessaires
\end{itemize}

\subsection{Lancement}
\begin{verbatim}
git clone https://github.com/...
cd destiny-raid-companion
cp .env.example .env  # et éditer avec sa clé API
docker-compose up -d
\end{verbatim}

\subsection{Arrêt}
\begin{verbatim}
docker-compose down
\end{verbatim}

\section{Ce que j'ai appris}

\begin{itemize}
    \item Automatiser, c'est gagner du temps sur le long terme. Ce qui est long au début (écrire les Dockerfile, configurer la CI) devient un gain énorme ensuite.
    \item Docker m'a sauvé plusieurs fois : "Ça marche sur ma machine" n'est plus une excuse, ça marche pareil partout.
    \item La CI m'oblige à être discipliné : je ne peux pas tricher et pousser du code qui ne passe pas les tests.
    \item Documenter le déploiement, c'est aussi pour moi dans 6 mois quand j'aurai oublié comment ça marche.
\end{itemize}
